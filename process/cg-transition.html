How to transition work from a Community Group to a Working Group

This Guidebook is the collected wisdom of the W3C Group Chairs and other collaborators.

Also On This Page →

This page concerns itself only with generically applicable requirements when transitioning work from a Community Group to a Working Group. Specific Community Groups and Working Groups may have additional specific requirements.

To assess whether a given work is ready to transition to a Working Group, please refer to W3C Recommendation Track Readiness Best Practices.

Figuring out how the Community Group/Working Group will collaborate

When work transitions from a Community Group to a Working Group, it is often the case that the Community Group will want to be in a position to be involved in the discussions subsequent to the transition, for the said specific work or for some related work.

Getting approval to transition

  • Has the Community Group agreed to the transition in a manner consistent with its practices? Is there clarity on how the Community Group will collaborate with the Working Group in the future (if at all)?
  • Does the Working Group that would take over the work exists? If not, a charter needs to be written and approved for the new Working Group to be created.
  • If the Working Group exists, is the work being transitioned in the scope of its existing charter? If not, the target Working Group needs to be rechartered to accommodate the new work.

Operationalizing the transition

IPR considerations:

  • Have all contributions incorporated in the work being transitioned been associated with a Community Group Contribution Licensing Agreeement?
  • Has there been a call for Final Specification Agreement been made on the proposed work?
  • Are major contributors to the work in the Community Group expecting to join the relevant Working Group?

Participation:

  • Make sure to keep the Community Group participants aware of the progress on the transition roadmap, in particular if it involves chartering or re-chartering a Working Group
  • If Community Group participants want to join the Working Group, but are not affiliated with a W3C Member organization, please invite them to contact Alan Bird to explore W3C Membership - during that exploration, Community Group participants may continue participating in the Working Group as Invited Experts

Assuming the work has been developed in a github repository:

  • If the repository is hosted in a Github organization not yet used by W3C to host Recommendation work:
    • Will the repository itself be transitioned to the W3C GitHub organization?
    • If not, is the repository owner willing to grant admin rights to @w3cbot (which is used to help with managing operational matters on W3C work, incl for horizontal review and IPR management)?
  • Is the repository already being monitored by W3C's IPR Repo Manager?
    • If not, evaluate even more carefully the IPR considerations set above.
    • If yes, once the transition is enacted (e.g. when the new Working Group is launched), the IPR manager will need to be updated to reflect the new IPR status of the work (contact dom@w3.org or plh@w3.org); if the repo is being moved or renamed at the time of transition, please coordinate specifically with dom@w3.org as this operation requires special handling at the moment.
  • Other repository administrativia:
    • If not already set up, ask plh@w3.org to configure the repository with the labels used for tracking horizontal reviews
    • If not already set up, make sure members of the horizontal-admin team have triage access to the repo; if unsure, ask plh@w3.org
    • Update or add LICENSE.md, CONTRIBUTING.md, CODE_OF_CONDUCT.md to reflect the new contribution framework
    • If not already set up that way, consider moving away from using master as the name of the default branch
